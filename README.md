# How to Convert PPK to id_rsa: The Complete SSH Key Conversion Guide for VPS Users

So you've got a `.ppk` file sitting on your desktop, and your terminal is throwing a `permission denied (publickey)` error every time you try to SSH into your server. Welcome to the club — this is one of those classic "why is this so confusing" moments that every developer hits at least once.

The short version: PuTTY and OpenSSH speak slightly different dialects of the same language. Your `.ppk` file is PuTTY's format; Linux servers and tools like `ssh`, `git`, and most CI/CD pipelines expect OpenSSH format (`id_rsa`). Converting between the two takes about two minutes once you know what you're doing.

This guide walks you through every method — command line, GUI, and a few edge cases — so you can pick whatever fits your situation.

---

## Why This Conversion Is Even Necessary

Here's the context that most quick tutorials skip.

SSH keys are not a single universal thing. The *cryptographic math* underneath is identical — RSA is RSA, Ed25519 is Ed25519. But the way those keys get *serialized into a file* differs between implementations. PuTTY uses its own `.ppk` container format (currently version 3), while OpenSSH uses a PEM-based format that starts with `-----BEGIN OPENSSH PRIVATE KEY-----` or `-----BEGIN RSA PRIVATE KEY-----`.

When you generate a key pair inside PuTTYgen on Windows and then try to use that `.ppk` file with `ssh -i` on Linux, the tool simply doesn't understand the format. It's like trying to open a `.docx` in Notepad — the bytes are there, they just aren't arranged the way the reader expects.

Common scenarios where you'll hit this:

- **Provisioning a Linux VPS**: You generated keys on Windows with PuTTY, now you need them on a fresh server.
- **Using Git over SSH**: GitHub, GitLab, and Bitbucket all expect OpenSSH public key format in authorized_keys.
- **CI/CD pipelines**: GitHub Actions, GitLab CI, and Jenkins agents all run `ssh` under the hood, not PuTTY.
- **Connecting via terminal on macOS/Linux**: The native `ssh` client ignores `.ppk` files entirely.

## What You Need Before Starting

- Your `.ppk` private key file (generated by PuTTYgen)
- If the key has a passphrase, you'll need it
- One of the following: PuTTYgen (GUI), `puttygen` CLI tool, or `ssh-keygen`

That's it. No paid tools, no magic software.

---

## Method 1: Command Line with `puttygen` (Linux / macOS / WSL)

This is the cleanest and most scriptable approach. If you're on Linux or macOS — or using WSL on Windows — this is the recommended method.

### Step 1: Install putty-tools

**Debian / Ubuntu:**
bash
sudo apt-get update && sudo apt-get install putty-tools


**Fedora / RHEL / CentOS:**
bash
sudo dnf install putty


**macOS (Homebrew):**
bash
brew install putty


### Step 2: Convert the Private Key

bash
puttygen your-key.ppk -O private-openssh -o ~/.ssh/id_rsa


If your key is passphrase-protected, `puttygen` will prompt you for it.

### Step 3: Extract the Public Key

bash
puttygen your-key.ppk -O public-openssh -o ~/.ssh/id_rsa.pub


### Step 4: Fix Permissions

This step is non-optional. OpenSSH *refuses* to use a private key file that has loose permissions:

bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_rsa
chmod 644 ~/.ssh/id_rsa.pub


### Step 5: Test the Connection

bash
ssh -i ~/.ssh/id_rsa username@your-server-ip


If that works, you're done. If you get `WARNING: UNPROTECTED PRIVATE KEY FILE!`, double-check the `chmod 600` step.

## Method 2: PuTTYgen GUI on Windows

For Windows users who aren't using WSL, the graphical PuTTYgen tool handles this without any command-line work.

1. **Open PuTTYgen** (download it from the official PuTTY page if you don't have it)
2. Click **"Load"** and select your `.ppk` file
3. Enter the passphrase if prompted
4. Go to **Conversions** → **Export OpenSSH key**
5. Save the file — name it `id_rsa` for standard SSH client compatibility
6. If it warns you about saving without a passphrase, that's fine for most use cases

Now transfer this file to your Linux system (via SFTP, SCP, or just copy-pasting the contents) into `~/.ssh/id_rsa`, then set permissions as shown in Method 1.

> **Heads up**: Modern PuTTYgen (version 0.75+) exports in the newer OpenSSH format by default. If you need the older PEM format (`-----BEGIN RSA PRIVATE KEY-----`) for compatibility with older tools, use **Conversions → Export OpenSSH key (force new file format)** or the older version of PuTTYgen.

---

## Method 3: Using `ssh-keygen` for Format Conversion

If you already have a PEM-format private key (not a `.ppk`, but something that looks like `-----BEGIN RSA PRIVATE KEY-----`), `ssh-keygen` can re-format it:

bash
ssh-keygen -p -m PEM -f ~/.ssh/id_rsa


This converts an existing OpenSSH new-format key to the older PEM format. It's less commonly needed but occasionally useful when working with older SSH servers or legacy tools.

To convert a key to the newer OpenSSH format from PEM:

bash
ssh-keygen -p -m OpenSSH -f ~/.ssh/id_rsa


Note that `ssh-keygen` **cannot** directly convert `.ppk` files — you still need `puttygen` for that initial step.

---

## Method 4: The Reverse — Converting id_rsa to PPK

Sometimes you need to go the other direction: you have an OpenSSH key and need to use it with PuTTY on Windows.

**Command line:**
bash
puttygen id_rsa -o output-key.ppk


**GUI:**
1. Open PuTTYgen
2. Go to **Conversions → Import key**
3. Select your `id_rsa` file
4. Click **Save private key**

This is useful when a cloud provider gives you a `.pem` file (common with AWS EC2) and you want to use it with PuTTY.

## Troubleshooting Common Errors

### "Invalid key" or "Could not load this key"

- Make sure you're pointing at the right file — `.ppk` for input, not the public key
- Try re-exporting with PuTTYgen GUI if the command-line fails

### "Permission denied (publickey)"

This is usually one of three things:
1. Wrong permissions on `~/.ssh/id_rsa` — run `chmod 600 ~/.ssh/id_rsa`
2. The public key isn't in `~/.ssh/authorized_keys` on the server
3. You're using the wrong key for that server — check with `ssh -i /path/to/specific/id_rsa`

### "WARNING: UNPROTECTED PRIVATE KEY FILE!"

File permissions are too open. Fix:
bash
chmod 600 ~/.ssh/id_rsa


### Key format mismatch with older servers

Some older SSH servers don't support the newer OpenSSH key format. Force PuTTYgen to export in classic PEM format, or use `ssh-keygen -p -m PEM -f id_rsa` to convert.

## Setting Up SSH Keys on a Fresh VPS

Once your key is in OpenSSH format, here's the standard workflow for adding it to a new server.

### If You Have Password Access First

bash
# Copy your public key to the server
ssh-copy-id -i ~/.ssh/id_rsa.pub username@server-ip

# Or manually
cat ~/.ssh/id_rsa.pub | ssh username@server-ip "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"


### Verify Key-Based Login Works

bash
ssh -i ~/.ssh/id_rsa username@server-ip


Once confirmed, you can disable password authentication in `/etc/ssh/sshd_config` for better security:


PasswordAuthentication no
PubkeyAuthentication yes


Then restart SSH: `sudo systemctl restart sshd`

## Connecting to DMIT (or Any VPS) After Key Conversion

Here's the practical end-to-end workflow, assuming you've just provisioned a fresh DMIT VPS and you're working from Windows with a `.ppk` key you generated earlier.

**1. Convert your `.ppk` to `id_rsa`** using PuTTYgen GUI or `puttygen` CLI as described above.

**2. Transfer your public key to the server** — since this is a fresh server, you'll likely use the root password provided in the DMIT control panel for the first login:

bash
ssh-copy-id -i ~/.ssh/id_rsa.pub root@<your-dmit-ip>


**3. Test key-based login:**

bash
ssh -i ~/.ssh/id_rsa root@<your-dmit-ip>


**4. Create a non-root user and set up their authorized_keys:**

bash
adduser yourusername
mkdir -p /home/yourusername/.ssh
cp ~/.ssh/authorized_keys /home/yourusername/.ssh/
chown -R yourusername:yourusername /home/yourusername/.ssh
chmod 700 /home/yourusername/.ssh
chmod 600 /home/yourusername/.ssh/authorized_keys


**5. Harden SSH configuration** in `/etc/ssh/sshd_config`:


PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes


**6. Restart SSH:** `systemctl restart sshd`

From this point, all SSH access goes through your converted key. No passwords flying over the wire.

## Quick Reference: puttygen Command Cheat Sheet

bash
# PPK → OpenSSH private key
puttygen key.ppk -O private-openssh -o id_rsa

# PPK → OpenSSH public key
puttygen key.ppk -O public-openssh -o id_rsa.pub

# OpenSSH → PPK
puttygen id_rsa -o key.ppk

# PPK → PEM (older format, maximum compatibility)
puttygen key.ppk -O private-openssh-new -o id_rsa_new

# Change passphrase on a PPK
puttygen key.ppk --old-passphrase oldpass --new-passphrase newpass -o key-new.ppk

# Show key fingerprint
puttygen key.ppk -O fingerprint
